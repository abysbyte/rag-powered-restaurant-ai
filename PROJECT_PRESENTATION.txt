PROBLEM STATEMENT & SOLUTION
================================================================================

PROBLEM:
--------
Restaurant review data is scattered across multiple sources and contains valuable 
information. Extracting meaningful insights from this unstructured data manually is 
time-consuming and inefficient. Users need an intelligent way to query restaurant 
data using natural language and receive context-aware, accurate responses.

SOLUTION:
---------
PIZZA(AI)LAB implements a Retrieval-Augmented Generation (RAG) system that:

1. Processes restaurant review data into semantic vector embeddings
2. Stores embeddings in a vector database (Chroma)
3. Retrieves relevant reviews based on user queries
4. Feeds retrieved context to an LLM for intelligent response generation
5. Presents answers through a modern, user-friendly web interface

This approach ensures responses are grounded in actual data while being 
intelligently synthesized by an LLM, avoiding hallucinations.

DETAILED WORKFLOW - END TO END
================================================================================

PHASE 1: USER AUTHENTICATION & ONBOARDING
------------------------------------------

Step 1: User Signup
├─ User enters email, username, password in React form
├─ Frontend validates input
├─ HTTP POST to /auth/signup endpoint
├─ Backend receives credentials
├─ Password hashed using bcrypt (salt + hash)
├─ User record created in SQLite database
├─ Success response sent to frontend
└─ User redirected to login page

Step 2: User Login
├─ User enters email and password
├─ Frontend submits HTTP POST to /auth/login
├─ Backend verifies email exists and password matches
├─ JWT token generated with payload:
│   └─ user_id, email, exp (24 hours from now)
├─ Token signed with SECRET_KEY using HS256 algorithm
├─ Token returned in JSON response
├─ Frontend stores token in localStorage
└─ User redirected to home/chat page

Step 3: Token Persistence
├─ Token retrieved from localStorage on page load
├─ Token added to Authorization header of all requests
├─ Backend middleware verifies token before processing requests
└─ User session maintained across browser refresh


PHASE 2: QUERY PROCESSING & RAG PIPELINE
-----------------------------------------

Step 1: User Input
├─ User types restaurant-related question in chat interface
├─ Example: "What are customers saying about pizza quality?"
├─ User clicks send button
└─ Question transmitted to backend

Step 2: JWT Verification
├─ Frontend adds JWT token to request headers
├─ Backend receives HTTP POST to /query endpoint
├─ Middleware extracts and verifies JWT token
├─ If invalid or expired → 401 Unauthorized response
├─ If valid → proceed to next step
└─ Request passed to query handler

Step 3: Vector Embedding Generation
├─ User question received: "What are customers saying about pizza quality?"
├─ Question passed to Ollama embedding model (mxbai-embed-large)
├─ Model converts text to 384-dimensional dense vector
│   └─ Each dimension captures semantic meaning
├─ Vector representation captures semantic similarity
└─ Embedding ready for database search

Step 4: Semantic Search in Vector Database
├─ Query embedding sent to Chroma vector store
├─ Chroma computes similarity scores:
│   └─ Cosine similarity between query vector and stored review vectors
├─ Top 3 most relevant reviews retrieved (k=3)
│   └─ Example results:
│       1. "Pizza quality is consistently excellent" (Rating: 5.0)
│       2. "Best pizza I've had, crispy crust" (Rating: 4.8)
│       3. "Pizza dough quality excellent" (Rating: 4.7)
├─ Reviews include metadata (rating, date, title, content)
└─ Retrieved reviews passed to next step

Step 5: Context Preparation
├─ Retrieved reviews formatted into readable context string
├─ Context includes:
│   ├─ Review titles
│   ├─ Full review text
│   └─ Rating information
├─ Example context:
│   "Review 1: 'Pizza quality is consistently excellent' - 5 stars
│    Review 2: 'Best pizza I've had, crispy crust' - 4.8 stars
│    Review 3: 'Pizza dough quality excellent' - 4.7 stars"
└─ Context ready for LLM

Step 6: LLM Prompt Template
├─ LangChain prompt template invoked:
│
│   Template:
│   "You are an expert in answering questions about a pizza restaurant.
│    Here are some relevant reviews: {reviews}
│    Here is the question to answer: {question}"
│
├─ Template variables filled:
│   ├─ {reviews} ← Retrieved review context
│   └─ {question} ← Original user question
│
├─ Final prompt created:
│   "You are an expert in answering questions about a pizza restaurant.
│    Here are some relevant reviews: 
│    [3 reviews with ratings and content]
│    Here is the question to answer: What are customers saying about 
│    pizza quality?"
│
└─ Prompt ready for LLM inference

Step 7: Ollama LLM Processing
├─ Prompt sent to Ollama LLM (llama3.2:1b model)
├─ LLM processes prompt and context
├─ Model generates response token-by-token
│   └─ Each token predicted based on previous tokens and context
├─ Response example:
│   "Based on the customer reviews, pizza quality at this restaurant
│    is highly regarded. Customers consistently praise the pizza for
│    its excellent quality, with ratings averaging 4.8 stars. 
│    Customers specifically mention the crispy crust and quality dough
│    as standout features. The reviews suggest consistently high quality
│    across different pizza orders."
└─ Complete response generated

Step 8: Response Formatting
├─ Generated response collected from LLM
├─ Response formatted as JSON:
│   {
│     "answer": "[LLM response text]",
│     "retrieved_reviews": [list of 3 reviews],
│     "status": "success",
│     "timestamp": "[current time]"
│   }
├─ Metadata included for frontend display
└─ Response ready for transmission

Step 9: Frontend Display
├─ Frontend receives JSON response
├─ Response parsed and extracted
├─ Message components updated in React
├─ Chat interface updated:
│   ├─ User question displayed in chat
│   └─ AI response displayed below with special styling
├─ Message added to chat history
├─ UI scrolls to show latest message
└─ User sees intelligent, contextual response

TECHNICAL DECISIONS & RATIONALE
================================================================================

1. LOCAL LLM CHOICE (Ollama) - Why not OpenAI/Claude APIs?
   ✓ Cost-effective: No API costs
   ✓ Privacy: Data stays local
   ✓ Development: Works offline, no internet required
   ✓ Deployment: No external dependencies
   ✓ llama3.2:1b: Lightweight yet capable (1B parameters)
   
   Trade-off: Slightly lower quality than larger models
   Solution: Mitigated by RAG - context from actual data

2. VECTOR DATABASE CHOICE (Chroma) - Why not Pinecone/Weaviate?
   ✓ Local/self-hosted: No external services
   ✓ Embedded: Works in-process
   ✓ Lightweight: Minimal dependencies
   ✓ SQLite backend: Persistent, easy to manage
   ✓ Persistent storage: Data survives restarts
   
   Trade-off: Limited scalability vs cloud solutions
   Perfect for: Single-user/small deployments

3. LANGCHAIN ORCHESTRATION
   ✓ Abstracts complexity of RAG pipeline
   ✓ Modular components (Prompts, Models, Retrievers)
   ✓ Easy to swap components
   ✓ Built-in integration with Ollama/Chroma
   ✓ Standardized prompt templating
   
   Benefits: Focus on application logic, not plumbing

4. FASTAPI FRAMEWORK
   ✓ High performance (async support)
   ✓ Automatic API documentation (Swagger/ReDoc)
   ✓ Type hints support (Pydantic models)
   ✓ Built-in CORS support
   ✓ Modern, actively maintained
   ✓ Easy JWT integration
   
   Alternative considered: Flask (more familiar, less performant)

5. JWT AUTHENTICATION
   ✓ Stateless: Server doesn't need session storage
   ✓ Scalable: No session database needed
   ✓ Secure: Cryptographically signed tokens
   ✓ Expirable: 24-hour token lifespan
   ✓ Frontend-friendly: Easy to store in localStorage
   
   Trade-off: No server-side token revocation
   Solution: Accept for this use case

6. VITE + REACT FRONTEND
   ✓ Vite: Extremely fast build tool (instant HMR)
   ✓ React: Component-based, familiar to most developers
   ✓ Modern tooling: ESM support, better tree-shaking
   ✓ Development speed: Rapid iteration
   
   Alternative considered: Next.js (adds complexity)

7. SQLite DATABASE
   ✓ Simple setup: No external database needed
   ✓ File-based: Easy backup and portability
   ✓ Sufficient for: User auth storage (small scale)
   ✓ No configuration: Works immediately
   ✓ SQLAlchemy ORM: Type-safe queries

INTERVIEW TALKING POINTS
================================================================================

1. PROBLEM SOLVING
"This project demonstrates how to solve the problem of extracting meaningful
insights from unstructured restaurant review data. Traditional keyword-based
search fails because 'excellent quality' and 'great crust' have different
keywords but similar meaning. By using vector embeddings and semantic search,
we can find contextually similar reviews regardless of exact wording."

2. TECHNICAL DEPTH
"The application implements a complete RAG (Retrieval-Augmented Generation)
pipeline. User questions are converted to embeddings, similar reviews retrieved
from a vector database, and this context is provided to an LLM to generate
grounded, accurate responses. This avoids hallucinations because answers are
based on real review data."

3. SYSTEM DESIGN
"The architecture separates concerns: FastAPI handles business logic and API,
SQLite stores user data, Chroma manages vector embeddings, and Ollama provides
local LLM capabilities. Frontend and backend communicate via JWT-authenticated
REST APIs. This modular design allows each component to be swapped independently."

4. SECURITY CONSIDERATION
"Authentication uses JWT tokens with 24-hour expiration. Passwords are hashed
with bcrypt before storage, so even database breach doesn't expose credentials.
Token verification middleware ensures all protected endpoints validate auth.
CORS is configured to allow frontend requests specifically."

5. PERFORMANCE OPTIMIZATION
"The vector store is cached to disk after first-run embedding generation.
This means subsequent application startups are instant (~100ms) instead of
2-5 minutes. FastAPI's async support handles concurrent requests efficiently.
Local LLM eliminates network latency compared to API-based alternatives."

6. SCALABILITY CONSIDERATIONS
"For production scaling, I would: upgrade SQLite to PostgreSQL, implement
rate limiting to prevent abuse, add caching layers, potentially shard vector
store, and deploy to Kubernetes. Current design works well for single-user
or small-team deployments."

7. CHALLENGE RESOLUTION
"One challenge was handling first-run setup (embedding generation takes time).
I solved this by implementing conditional logic to only generate embeddings
once and cache them. Another was ensuring security with JWT tokens while
maintaining developer experience - the 24-hour expiration is appropriate for
a demo while still being testable during development."
